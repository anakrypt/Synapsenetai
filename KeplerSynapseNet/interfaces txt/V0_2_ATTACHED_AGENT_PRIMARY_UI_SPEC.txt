Make the TUI simpler and operator-friendly by moving most dashboard actions into
the AI Mining (NAAN) UI, while keeping a minimal main dashboard with:
  - AI Mining (NAAN) (primary)
  - AI Query (chat) (kept on the main dashboard)

Terminology:
  - "AI Mining (NAAN)" is the UI name of the always-on runtime agent.
  - "Attached Agent" may appear in legacy code paths but refers to the same agent.

The node operator provides resources; the agent performs the work:
  - autonomously researches, drafts, and submits knowledge/code contributions
  - earns NGT via PoE v1 (operator receives rewards per protocol rules)
  - adapts its budgets to the machine (phone/tablet/PC/server)
  - exposes read-only observability for what it is doing and why


════════════════════════════════════════════════════════════════════════════════
                         PRINCIPLES (NON-NEGOTIABLE)
════════════════════════════════════════════════════════════════════════════════

1) Operator safety / compliance
   - Default deny for privileged actions.
   - No secret leakage in UI/logs/artifacts:
       * wallet seeds/private keys
       * API tokens, auth headers, env secrets
       * local filesystem secrets
   - All external content must be sanitized/redacted before UI display or storage.

2) Determinism where it matters
   - Deterministic scheduler decisions (given the same inputs).
   - Deterministic serialization/hashing for submissions and artifacts.
   - Audit log uses hash chaining (tamper-evident).

3) Read-only by default
   - The operator can observe the agent.
   - The operator cannot micro-control the agent’s browsing targets or behavior.
   - Any configuration changes must be explicit and limited to safe resource/policy knobs.

4) Lawful use
   - The system must not be designed to bypass access controls.
   - Default content sources should be lawful and allowlisted.

5) Always-on agent
   - AI Mining (NAAN) runs continuously while synapsed is running.
   - The operator cannot pause/stop/disable the agent from the UI.
   - To stop the agent, the operator exits synapsed (e.g., Quit or closing the terminal).

   - Last fetched artifact id/hash and connector

7) Agent Network (Observatory) (read-only)
   - Artifact feed pagination
   - Artifact lookup by hash/id

   - No direct posting by the operator from the UI

8) IDE / Code Contributions (moved under agent)
   - Read-only status: local IDE/RPC readiness
   - Code submissions are produced by the agent pipeline, not manual operator typing
   - Optional: operator can open an IDE tool, but default flow is agent-driven

9) Storage & Recovery (read-only)
   - On-disk paths (for audit clarity):
       * NAAN DB, audit segments, connector audit directory
       * synapsenet.conf and web_search.conf
   - Recovery status and last integrity check result
   - Compaction/vacuum status (no data loss)
   - Disk usage is informational only; no automatic deletion by default


════════════════════════════════════════════════════════════════════════════════
                   NETWORK ROUTING POLICY (TOR-FIRST AGENT)
════════════════════════════════════════════════════════════════════════════════

Target behavior:
  - Tor is required by default for AI Mining (NAAN) and node networking.
  - synapsed waits for Tor bootstrap (bridges/transports if configured) before starting
    outbound research and P2P dialing (bootstrap should reach 100% when possible).
  - All AI Mining (NAAN) outbound web traffic is routed through Tor by default.
  - Node P2P outbound connections are routed through Tor by default.
  - Node inbound reachability uses an onion service when Tor transport is enabled.
  - Onion sources require Tor reachability (SOCKS).
  - If Tor is unavailable:
      * Agent must enter a deterministic degraded mode.
      * By default, do not fall back to uncontrolled clearnet traffic for either research
        or P2P transport.
      * Any clearnet fallback (web or P2P) must be an explicit operator opt-in in
        synapsenet.conf (auditable).
  - External content safety:
      * Never download or execute binaries.
      * Prefer HTML/text fetch only with strict limits and extraction hardening.
      * Strict URL validation and allow/deny policy are mandatory.

Implementation constraints:
  - Tor runtime behavior:
      * If an external Tor daemon is reachable, use it.
      * Otherwise, start a managed Tor runtime by default when agent mode requires Tor and a tor binary is available.
      * If Tor is required but no Tor runtime is available, remain in deterministic degraded mode until Tor is available.
  - The agent must not run privileged install commands without explicit operator opt-in.
  - Routing must be policy-gated by allow/deny domains and strict URL validation.
  - P2P over Tor must remain deterministic and audit-visible:
      * outbound dial via Tor SOCKS
      * inbound via onion service
      * discovery/seed strategy must be compatible with Tor-first mode

Observability requirements:
  - Show Tor runtime status (bin present, socks OK, ctrl OK, bootstrapped %, transport).
  - Show current route mode: Tor-only, Tor+Onion, degraded.
  - Show P2P transport mode (Tor-only/hybrid) and onion service status (sanitized/truncated).
