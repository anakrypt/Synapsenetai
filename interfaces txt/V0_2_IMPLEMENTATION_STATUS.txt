════════════════════════════════════════════════════════════════════════════════
                         QUANTUM SECURITY STATUS
════════════════════════════════════════════════════════════════════════════════

Updated on February 23, 2026.

Implemented in code:
  • CLI now accepts `--security quantum-ready` and maps it to `SecurityLevel::QUANTUM_READY`.
  • Config defaults now include `security.quantum_enabled=true` and `security.level=high`.
  • Runtime config loading now honors CLI precedence and applies config defaults for quantum enablement and security level.
  • `QuantumManager` now keeps persistent hybrid KEM and hybrid signature keypairs instead of generating ephemeral signing keys per call.
  • `QuantumManager` now enforces non-classic KEM/signature policy when level is not `standard`.
  • Kyber shared-secret derivation is now symmetric between encapsulation and decapsulation paths.
  • Dilithium and SPHINCS+ signatures are now deterministic and verifiable against public keys.
  • Hybrid signature verification now validates both the classic signature segment and the Dilithium segment.
  • Quantum payload framing now supports explicit mode signaling (hybrid or QKD) with backward-compatible decode handling.
  • Quantum-ready mode now attempts QKD session establishment and falls back to hybrid KEM when QKD material is unavailable.
  • Runtime quantum status now exposes selected algorithms and QKD/hybrid operation counters.
  • `CryptoSelector` algorithm availability checks are now mutex-safe across selection and listing calls.
  • `node.status` now includes a structured `quantum` status object with:
      - `enabled`, `requestedLevel`, `effectiveLevel`, `managerInitialized`, `isQuantumSafe`
      - `qkdConnected`, `qkdSessionActive`
      - `selectedKEM`, `selectedSignature`, `selectedEncryption`
      - `qkdEncryptOperations`, `hybridEncryptOperations`, `qkdDecryptOperations`, `hybridDecryptOperations`, `qkdFallbackDecryptOperations`
      - `counters` (`hybridOperations`, `qkdSessionsEstablished`, and PQC signature/KEM counters)

Operational behavior:
  • Default runtime behavior is quantum-enabled with `high` policy unless overridden by CLI/config.
  • `standard`: quantum subsystem is disabled unless explicitly enabled.
  • `high` / `paranoid`: hybrid quantum-safe encryption and signature paths are active.
  • `quantum-ready`: QKD is used when available; hybrid fallback remains active for graceful degradation.
  • Downgrade boundary is explicit: decrypting a QKD-framed payload after runtime downgrade from `quantum-ready` fails closed and increments `qkdFallbackDecryptOperations`.

════════════════════════════════════════════════════════════════════════════════
                       POLICY + CONNECTOR STATUS MODEL
════════════════════════════════════════════════════════════════════════════════

Connectors:
  • Clearnet: Tor-routed fetch in Tor-required mode
  • Tor: clearnet fetch routed through Tor SOCKS
  • Onion: .onion fetch via Tor

Defaults:
  • `agent.tor.required=true` by default
  • `web.inject.enabled=true`, `web.inject.onion=true`, and `web.inject.tor_clearnet=true` by default
  • `agent.routing.allow_clearnet_fallback=false` and `agent.routing.allow_p2p_clearnet_fallback=false` by default
  • Tor-required mode enables SOCKS proxy routing for outbound P2P and web connectors

Safety behavior:
  • Tor-required mode uses fail-closed policy when Tor is unavailable (no uncontrolled clearnet fallback)
  • Remote model rent/infer RPC paths fail closed with `tor_fail_closed_remote_model` when Tor-required route policy blocks outbound P2P discovery
  • Route-policy rejections do not trigger score penalties
  • Connector health transitions are explicit (ONLINE/DEGRADED/OFFLINE/POLICY_BLOCKED)
  • Extraction risk gate blocks unsafe outputs with deterministic reasons

Status surfaces:
  • `node.status` includes Tor route fields: `torRequired`, `torReachable`, `torManaged`, `torDegraded`, `routeMode`, fallback flags, and route allow flags
  • `node.status` and `naan.status` include `networkHealth` with pressure, overload, buffered RX bytes, rejection/eviction/temp-ban totals, malformed/rate-limited totals, backpressure counters, gossip suppression/subset routing counters, and consensus lag
  • `node.status` includes `quantum` runtime status with effective policy, selected algorithms, and QKD/hybrid operation counters
  • `naan.status` includes matching route fields plus deterministic reward visibility counters:
      - `agentRewardObservedSubmissions`
      - `agentRewardFinalizedSubmissions`
      - `agentRewardCreditedSubmissions`
      - `agentRewardExpectedAtoms`
      - `agentRewardCreditedAtoms`


════════════════════════════════════════════════════════════════════════════════
                    NETWORK SCALE + OVERLOAD PROTECTION
════════════════════════════════════════════════════════════════════════════════

Implemented controls:
  • Adaptive inbound admission with deterministic eviction of lower-priority peers under pressure
  • Adaptive outbound admission with bounded `maxOutboundConnections` and subnet/IP quota checks
  • Per-IP and per-subnet concurrent connection limits to reduce concentration and Sybil pressure
  • Per-peer token-bucket message throttling with deterministic refill and consume semantics
  • Malformed payload tracking with decay; temporary bans use bounded decay-based durations
  • Overload mode enters/exits based on deterministic peer/message pressure thresholds
  • Inventory fanout compression path under heavy fanout (subset routing + suppression counters)
  • `getdata` request/serve caps when overloaded to avoid amplification collapse
  • Deterministic bounded vote dedup cache for PoE vote handling

Bootstrap discovery hardening:
  • Bootstrap nodes carry health metadata (`healthScore`, `lastFailureAt`, `quarantineUntil`)
  • Seed failures quarantine entries for a bounded TTL; healthy seeds rotate first
  • `node.seeds` and `node.discovery.stats` expose active vs quarantined seed state
