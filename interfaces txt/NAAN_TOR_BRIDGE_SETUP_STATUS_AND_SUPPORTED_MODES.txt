2. SUPPORTED TOR RUNTIME MODES
==================================================

Mode: `agent.tor.mode=auto`
- Behavior: SynapseNet may use existing SOCKS if reachable, and may autostart managed Tor if Tor is required and unavailable.
- Default managed target ports are typically `9050/9051` unless config overrides.
- Use when operator does not need strict separation between managed/external runtime.

Mode: `agent.tor.mode=managed`
- Behavior: SynapseNet is allowed to start and own a Tor process.
- Recommended only when operator is not already running a bridge-configured external Tor.
- SynapseNet tracks managed PID and attempts safe owned shutdown on exit.

Mode: `agent.tor.mode=external`
- Behavior: SynapseNet is SOCKS client only; managed Tor autostart is disabled.
- Recommended for bridge workflows (obfs4) and shared setup with Tor Browser + CLI tools.
- Typical config: SOCKS `127.0.0.1:9150` with optional ControlPort `9151`.

==================================================
3. RECOMMENDED BRIDGE WORKFLOW (EXTERNAL TOR)
==================================================

Recommended operator architecture:
- One external Tor process (bridge-enabled) on `127.0.0.1:9150`
- SynapseNet uses `agent.tor.mode=external`
- Tor Browser may reuse the same SOCKS5 `127.0.0.1:9150`
- CLI tools (`curl`, `w3m`, etc.) may use the same SOCKS5 port

Why this is recommended:
- Avoids `9050` / `9150` conflicts between SynapseNet-managed Tor and external bridge Tor
- Makes debugging easier (single Tor runtime)
- Aligns SynapseNet/Browser/CLI traffic path for validation

Bridge pool storage (SynapseNet runtime):
- Bridge file: `<DATA_DIR>/tor/bridges.obfs4.txt`
- Cache file: `<DATA_DIR>/tor/bridge_pool_cache.json`

NAAN site policy storage:
- Site policy file: `<DATA_DIR>/naan_agent_web.conf`
- Keys edited from TUI:
  - `clearnet_site_allowlist=...`
  - `onion_site_allowlist=...`

4. RUNTIME STATUS FIELDS (OPERATOR-RELEVANT)
==================================================

`naan.status` and `node.status` expose Tor diagnostics including:
- `torRuntimeMode`
- `torSocksPort`
- `torControlPort`
- `torControlReachable`
- `torReachable` / `torSocksReachable`
- `torReadyForWeb`
- `torBootstrapState`
- `torConflictHint9050`
- `torManaged`
- `torManagedPid`
- `torReadyForOnionService`
- `torOnionServiceActive`
- `torOnionServiceState`
- `torBridgeProvider`
- `torBridgeProviderUpdatedAt`
- `torBridgeCacheAgeSeconds`
- `torRoute` object with `runtimeMode`, `webReady`, `bootstrapState`, `conflictHint9050`

Operator interpretation:
- `torReachable=true` means SOCKS listener is reachable
- `torReadyForWeb=true` means real Tor SOCKS web probe succeeded
- `torBootstrapState=SOCKS_UP_BOOTSTRAP` usually means SOCKS is up but web circuits are not ready yet
- `torConflictHint9050=true` usually indicates SynapseNet expects `9050` while working external Tor is on `9150`
- `torControlReachable=false` with `torReadyForWeb=true` indicates web routing works but ControlPort-dependent onion service may remain unavailable
- `torOnionServiceState=CONTROL_UNREACHABLE_WEB_READY` indicates exactly that split mode (web-ready, onion-service not ready)
- `torBridgeCacheAgeSeconds` + `torBridgeProvider.reasonCode/error` help identify stale/expired bridge cache vs transport/bootstrap failures

7. KNOWN LIMITS / NEXT WORK
==================================================

- Bridge distribution is often interactive (CAPTCHA/JS) and should not be scraped.
  The runtime remote adapter is intentionally conservative: if a remote source looks
  interactive, it returns `manual_step_required` and asks the operator to provide
  bridge lines via file/manual config.
- GitHub Actions live external-9150 workflow is secret-gated and may be disabled in CI:
  - `.github/workflows/tor-external-9150-live-e2e.yml` requires `TOR_OBFS4_BRIDGES`
- Cross-platform smoke/validator scripts are operator tools; live network validation is not CI-executed in this workspace, but CI now enforces script syntax/help gates across Linux/macOS/Windows.
- Windows script behavior should be validated on a real Windows host with tor.exe + obfs4proxy.exe installed.
- Linux and Windows live external-validation evidence is still pending on real hosts.
- Site allowlist is operator-controlled and security-sensitive:
  - adding untrusted domains can expose the node to malicious content
  - AI/IDE triage is useful, but does not replace AV/EDR controls
- Latest macOS evidence confirms both managed lifecycle and external `9150` web-ready operation (`torReadyForWeb=true`, `torBootstrapState=WEB_READY`):

8. MINIMUM EXTERNAL TOR (9150) CONFIG FOR SYNAPSENET
==================================================

agent.tor.required=true
agent.tor.mode=external
agent.routing.allow_clearnet_fallback=false
agent.routing.allow_p2p_clearnet_fallback=false
agent.tor.socks_host=127.0.0.1
agent.tor.socks_port=9150
tor.socks.host=127.0.0.1
tor.socks.port=9150

Optional:
tor.control.port=9151

Tor Browser (macOS) external SOCKS client shortcut (no Tor ownership, uses the same external Tor on 9150):

TOR_PROVIDER=none TOR_SOCKS_HOST=127.0.0.1 TOR_SOCKS_PORT=9150 "/Applications/Tor Browser.app/Contents/MacOS/firefox" --new-instance

9. QUICK NO-CONFLICT COMMAND SEQUENCE (MACOS)
==================================================

Repository location is represented as `<repo-root>` on purpose (no machine-specific paths).

1) Stop extra Tor owners (optional but recommended before restart):

pkill -f "/Applications/Tor Browser.app/Contents/MacOS/Tor/tor" || true
pkill -f "/opt/homebrew/bin/tor" || true
sleep 1

2) Start one shared external Tor on 9150:

cd <repo-root>/KeplerSynapseNet
tools/macos_tor_obfs4_helper.sh \
  --bridges-file /tmp/bridges.txt \
  --socks-port 9150 \
  --control-port 9151 \
  --bootstrap-check \
  --bootstrap-attempts 6 \
  --bridge-subset-size 4 \
  --takeover-port-owner \
  --keep-running \
  --out /tmp/tor-obfs4-synapsenet.conf

3) Verify the shared Tor path:

lsof -nP -iTCP:9150 -sTCP:LISTEN
curl --socks5-hostname 127.0.0.1:9150 https://check.torproject.org/api/ip --max-time 30

4) Start SynapseNet with external Tor config:

cd <repo-root>/KeplerSynapseNet
TERM=xterm-256color ./build/synapsed -D /tmp/synapsenet_fresh --dev -c /tmp/synapsenet_external_9150.conf

5) Start Tor Browser as SOCKS client of the same Tor:

TOR_PROVIDER=none TOR_SOCKS_HOST=127.0.0.1 TOR_SOCKS_PORT=9150 "/Applications/Tor Browser.app/Contents/MacOS/firefox" --new-instance

Operational rules:
- Do not run plain `tor` manually after this sequence.
- If helper reports `9150 already in use` and curl returns `"IsTor":true`, shared Tor is already running; continue with SynapseNet + Tor Browser client mode.
